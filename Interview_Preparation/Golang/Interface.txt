🧩 Advanced Go Interface Q&A (spoken answers made for interviews)

(clear, conversational, confident)

1️⃣ What is an interface in Go?

Answer (spoken):
“Basically, an interface defines a set of methods, and any type that implements those methods automatically satisfies the interface — no keyword, no declaration, it’s implicit.”

2️⃣ How is Go’s interface different from Java or C# interfaces?

Answer (spoken):
“To be honest, Go uses implicit implementation. You don’t explicitly say ‘implements’. If your type has the methods, it satisfies the interface. This gives loose coupling and better modularity.”

3️⃣ What is dynamic dispatch in Go interfaces?

Answer (spoken):
“When you call a method through an interface, Go uses the interface’s dynamic type and method table to decide which implementation to run at runtime. That’s dynamic dispatch — similar concept to polymorphism in other languages.”

4️⃣ What are the two parts stored inside an interface value?

Answer (spoken):
“An interface value holds two things:

The dynamic type

The dynamic value (data)
This pair is often called the type and the value.”

5️⃣ Why does interface{} == nil sometimes not work? (VERY important interview question)

Answer (spoken):
“This happens because an interface is nil only when both type and value are nil.
If the interface has a dynamic type but the underlying value is nil, the interface is not nil.
That’s the famous typed-nil issue.”

6️⃣ Can you give a simple example of the typed-nil trap?

Answer (spoken):
“Sure: if a function returns (*User, error) and you return nil, nil, but your error is set inside an error interface like var err *MyErr = nil, then the interface has a type (*MyErr) but value nil — so it’s not nil. This leads to conditions failing.”

7️⃣ What is an empty interface (interface{})?

Answer (spoken):
“Empty interface means no methods. So every type satisfies it. Basically, it’s like ‘any’ — used for generic containers before Go generics came.”

8️⃣ Why should you avoid using empty interface too much?

Answer (spoken):
“Because it loses type safety. You need type assertions or reflection to get the real value back, which can cause runtime errors if not handled carefully.”

9️⃣ What is type assertion?

Answer (spoken):
“It’s a way to extract the underlying value from an interface. For example:
str, ok := i.(string)
It tries to convert the dynamic type to a concrete type. Use comma-ok to avoid panics.”

🔟 What is type switch?

Answer (spoken):
“A type switch checks dynamic type of an interface and matches the case. It’s cleaner than multiple type assertions and used for polymorphic behavior.”

1️⃣1️⃣ What is a method set in Go?

Answer (spoken):
“Method set tells which methods belong to a type.
For value types: only value receiver methods.
For pointer types: both pointer receiver and value receiver methods.”

1️⃣2️⃣ Why does this matter? (Interviewers love this)

Answer (spoken):
“It matters because a value type doesn't implement an interface if the interface requires methods with pointer receivers. But a pointer type can implement all methods.”

1️⃣3️⃣ Can you explain interface satisfaction with method sets?

Answer (spoken):
“Yes — if an interface expects a method with pointer receiver, only *T satisfies it, not T.
If an interface requires only value receiver methods, both T and *T satisfy it.
This confuses many beginners, so interviewers check it.”

1️⃣4️⃣ What is a zero value of an interface?

Answer (spoken):
“It’s nil. Both dynamic type and dynamic value are nil.”

1️⃣5️⃣ Can Go interfaces embed other interfaces?

Answer (spoken):
“Yes — embedding lets you compose interfaces.
For example, io.ReadWriter embeds both Reader and Writer.
It’s like interface inheritance but simpler.”

1️⃣6️⃣ What happens when you compare two interface values?

Answer (spoken):
“Two interface values are equal only if both their dynamic types and dynamic values are equal.
If the underlying values are uncomparable (like slices or maps), comparing will panic.”

1️⃣7️⃣ Can you store a nil pointer in an interface?

Answer (spoken):
“Yes, you can store a nil pointer, but the interface itself won’t be nil because it still carries the dynamic type.
This is the root cause of many bugs.”

1️⃣8️⃣ What is the difference between interface vs generics?

Answer (spoken):
“Interfaces give runtime polymorphism; generics give compile-time type parameters.
Interfaces are great when behavior is important.
Generics are great for reusable algorithms and collections.”

1️⃣9️⃣ What is io.Reader interface and why is it important?

Answer (spoken):
“It’s a foundational interface in Go standard library. Anything that can read bytes implements it.
Because it’s so small, it allows a lot of code reuse through composition.”

2️⃣0️⃣ When should you use interfaces in Go?

Answer (spoken):
“Use interfaces at the consumer side, not at the producer side.
Define interface where you need flexibility — don’t force types to implement unnecessary interfaces. Go promotes small interfaces like ‘do one thing well’.”

2️⃣1️⃣ What are common mistakes with interfaces?

Answer (spoken):
“Overusing empty interface, forgetting typed-nil behavior, returning concrete types instead of interfaces, and designing huge interfaces instead of small focused ones.”

2️⃣2️⃣ What is interface dispatch cost? Is it slow?

Answer (spoken):
“Interface calls have a tiny overhead since they go through method table lookup.
But practically it’s negligible — Go is optimized for this.”

2️⃣3️⃣ Can interfaces store functions?

Answer (spoken):
“Yes, because function is also a type.
Any function type with no methods satisfies interface{} or any matching interface.”

2️⃣4️⃣ Why does Go prefer composition over inheritance?

Answer (spoken):
“Because interfaces describe behavior, not structure.
Go avoids class-style inheritance to keep things simple and flexible — smaller parts that you compose.”

2️⃣5️⃣ Can a type implement multiple interfaces?

Answer (spoken):
“Absolutely — since it’s implicit, a single type can satisfy any number of interfaces simply by having the required methods.”

2️⃣6️⃣ What is the empty method interface {} used for?

Answer (spoken):
“Sometimes used for tagging — when you want to express that a type has no methods required but still want a type group. Rare, but it exists.”

2️⃣7️⃣ What is the difference between nil interface and nil pointer inside interface?

Answer (spoken):
“A nil interface has no type and no value.
A non-nil interface with nil pointer has a type (*T) but value nil.
This is the most common interview confusion point.”

2️⃣8️⃣ Can an interface contain fields?

Answer (spoken):
“No — interfaces only contain method signatures. No fields, no concrete data.”

2️⃣9️⃣ What does ‘Duck Typing’ mean in Go?

Answer (spoken):
“It means ‘if it quacks like a duck, it’s a duck’.
If your type has the required methods, it satisfies the interface — this is duck typing.”

3️⃣0️⃣ Why do interviewers love interface questions?

Answer (spoken):
“Because interface behavior, method sets, and typed nil are the places where even experienced Go developers get confused. If you answer these well, you stand out immediately.”
