🧩 Advanced Goroutine Q&A — (spoken style + fillers)
1️⃣ What is a goroutine?

Answer (spoken):
“Basically, a goroutine is a lightweight thread of execution managed by the Go runtime. You just write go fn() and it runs concurrently — the runtime schedules thousands of them efficiently.”

2️⃣ How does a goroutine differ from an OS thread?

Answer (spoken):
“Actually, goroutines are much lighter than OS threads. OS threads are expensive (memory and context switch cost), while goroutines are multiplexed by Go’s scheduler onto a small number of OS threads — so many goroutines share fewer threads.”

3️⃣ Why do goroutines consume less memory?

Answer (spoken):
“The idea is that goroutine stacks start very small (a few KB) and grow/shrink dynamically. So instead of reserving megabytes per thread, Go allocates little memory initially and expands only when needed — that’s why they’re memory-efficient.”

4️⃣ Why are goroutines fast?

Answer (spoken):
“To be honest, they’re fast because creation and scheduling are cheap: the Go runtime uses an M:N scheduler (many goroutines to fewer OS threads), tiny initial stacks, and efficient context-switching in user space. So spawning thousands is cheap compared to OS threads.”

5️⃣ What is the M:N scheduler?

Answer (spoken):
“Basically, Go maps many goroutines (M) onto fewer OS threads (N). The scheduler has G (goroutine), M (OS thread), and P (processor context) — G-M-P model — and it schedules Gs on Ps which are tied to Ms.”

6️⃣ What are G, M and P?

Answer (spoken):
“G is a goroutine, M is an OS thread, and P is a logical processor that holds runtime state and run queues. P lets the runtime control how many goroutines can execute simultaneously (GOMAXPROCS controls number of Ps).”

7️⃣ How does Go handle blocking syscalls?

Answer (spoken):
“If a goroutine makes a blocking syscall, the runtime can park that M and attach a new M to the P — so other goroutines can keep running. So syscalls don’t block the whole runtime.”

8️⃣ What is a channel and how does it work?

Answer (spoken):
“Channels are typed pipelines for goroutines to safely communicate. An unbuffered channel blocks sender until receiver is ready; a buffered channel stores up to N values and only blocks when full or empty — it’s a safe way to pass data and synchronize.”

9️⃣ What is select used for?

Answer (spoken):
“select is like switch for channels. It waits on multiple channel operations and runs whichever is ready. It’s handy for timeouts, multiplexing, and cancellation.”

🔟 How do you cancel goroutines?

Answer (spoken):
“We generally use context.Context — you pass a cancellable context to goroutines and check ctx.Done() to stop work gracefully. It’s the idiomatic cancellation pattern in Go.”

1️⃣1️⃣ What are common goroutine leak patterns and how to avoid them?

Answer (spoken):
“Leaked goroutines often happen when a goroutine blocks forever on a channel or never sees cancellation. To avoid leaks, always use contexts, close channels carefully, add timeouts, and signal termination (done channels).”

1️⃣2️⃣ How do you limit number of concurrent goroutines?

Answer (spoken):
“One common approach is a worker pool or semaphore pattern using buffered channels. For example, send tokens into a channel with capacity N — each goroutine acquires a token and returns it when done, limiting concurrency.”

1️⃣3️⃣ What is sync.WaitGroup for?

Answer (spoken):
“We use WaitGroup to wait until a group of goroutines finishes. You Add() the number, each goroutine Done()s when complete, and Wait() blocks until zero — simple and useful for coordination.”

1️⃣4️⃣ What is the Go memory model and why it matters for goroutines?

Answer (spoken):
“Basically, the memory model defines happen-before relationships. You must use synchronization (channels, mutexes, atomics) to ensure ordering. Without proper sync, you can have data races and undefined behavior.”

1️⃣5️⃣ How to detect data races?

Answer (spoken):
“Use the race detector: run go test -race. It instruments the code to find concurrent unsynchronized access — very helpful in interviews to mention.”

1️⃣6️⃣ How do channels compare to locks? When to use which?

Answer (spoken):
“Channels are good for passing data and ownership (message-passing). Locks (sync.Mutex) are good for protecting shared mutable state. So ‘channel for communication, mutex for shared memory’ is a good rule of thumb.”

1️⃣7️⃣ What is runtime.Gosched() and runtime.Goexit()?

Answer (spoken):
“Gosched() yields the processor allowing other goroutines to run. Goexit() terminates the current goroutine immediately. Both are low-level and rarely needed in normal code.”

1️⃣8️⃣ What about preemption and scheduling fairness?

Answer (spoken):
“Since Go 1.14, the runtime supports preemptive scheduling — long-running goroutines are preempted so the scheduler stays fair. But if a goroutine runs pure user CPU code without making calls, the runtime periodically preempts it.”

1️⃣9️⃣ How do goroutines interact with GC?

Answer (spoken):
“The garbage collector coordinates with goroutines; goroutines can be paused briefly for GC safepoints. Modern GC is concurrent and low pause, but you should avoid generating excessive allocations in hot paths.”

2️⃣0️⃣ Why sometimes goroutines are slower for CPU-bound work?

Answer (spoken):
“To be honest, goroutines don’t make CPU work faster by themselves — if your task is CPU-bound, the speed is limited by GOMAXPROCS (number of OS threads running user code). Spawning many goroutines for CPU-bound tasks can add overhead. For CPU-heavy jobs, use a fixed worker pool sized to CPU cores.”

2️⃣1️⃣ How does Go handle many goroutines sleeping (timers, IO)?

Answer (spoken):
“That’s where goroutines shine — sleeping goroutines consume tiny memory and no OS thread while blocked, so millions of sleeping goroutines are feasible, which is great for servers handling many idle connections.”

2️⃣2️⃣ What is runtime.NumGoroutine() useful for?

Answer (spoken):
“It’s a simple runtime metric to inspect number of live goroutines — useful for monitoring and detecting leaks.”

2️⃣3️⃣ How to profile goroutine issues?

Answer (spoken):
“Use pprof (heap, cpu) and go tool pprof, and use goroutine profiles to see stacks. Also runtime/trace and net/trace help find blocking and scheduling issues.”

2️⃣4️⃣ What are best practices for designing concurrent systems with goroutines?

Answer (spoken):
“Keep goroutines small, add timeouts and cancellation, avoid sharing mutable state without locks, favor message-passing for ownership transfer, limit concurrency with worker pools, and always handle errors and cleanup.”

2️⃣5️⃣ How do goroutines behave with cgo and external C libraries?

Answer (spoken):
“When you call C code (cgo) that blocks, the runtime may create an OS thread to avoid blocking others, but C code can still block the OS thread. Also care is needed with thread-local state; cgo can complicate scheduling.”

2️⃣6️⃣ What is an example short answer to ‘why goroutines vs Java threads’?

Answer (spoken):
“Basically, Java threads are backed by OS threads with large stacks (MBs), so they’re heavier. Goroutines have tiny, growable stacks and are scheduled by the Go runtime, so you can run many more concurrently with lower memory and scheduling overhead.”

2️⃣7️⃣ How do buffered channels work and when to use them?

Answer (spoken):
“Buffered channels store N values so producers don’t block until the buffer is full. Use them when you want some decoupling between producer and consumer throughput, but be careful with buffer size to avoid hidden backpressure.”

2️⃣8️⃣ What is sync/atomic vs sync.Mutex?

Answer (spoken):
“sync/atomic offers low-level atomic operations for simple counters or flags and is faster for small ops. Mutex provides general mutual exclusion for complex critical sections. Use atomic for simple cases, mutex for complex state.”

2️⃣9️⃣ How to implement timeout for channel operations?

Answer (spoken):
“You can use select with time.After — for example:
select { case v := <-ch: ... case <-time.After(time.Second): // timeout } — simple and idiomatic.”

3️⃣0️⃣ What are common interview pitfalls about goroutines you should mention?

Answer (spoken):
“To be honest, interviewers like hearing about goroutine leaks, race conditions, misusing channels (closing wrong channel), uncontrolled goroutine creation, and ignoring cancellation — so mention these and how you prevent them.”

1. What is Polymorphism in Go?

Polymorphism refers to the ability of different types to implement the same interface and
 provide their own specific behavior for the methods defined by that interface. 
 This allows us to treat those different types interchangeably when using the interface, 
 even though they might have different behaviors.

2.  What is Encapsulation in Go?
 concept: 
 Encapsulation is the concept of restricting access to certain components of an object and exposing only what is necessary.

 In Go, encapsulation is achieved using exported and unexported fields:

Exported fields start with an uppercase letter and can be accessed from outside the package.

Unexported fields start with a lowercase letter and can only be accessed within the same package.


3. Inheritance in Go

Q: How does inheritance work in Go?

Go does not support traditional class-based inheritance like languages such as Java or "C#". 
Instead, Go uses composition to achieve similar functionality. 
Instead of inheriting from a parent class, a type can embed another type, effectively gaining its methods and behaviors.


3. Abstraction in Go

**concept**
 Abstraction is the concept of hiding the internal implementation details of an object and exposing only the necessary information or functionality.

Q: What is Abstraction in Go?

 In Go, abstraction is mainly achieved through interfaces. Interfaces allow you to define a set of methods that types must implement,
  without exposing how those methods are implemented.

Abstraction
Polymorphism – Achieved via interfaces, where different types can implement the same interface and be treated interchangeably.

Encapsulation – Achieved through structs with exported and unexported fields, and interchangeably accessor methods (getters/setters).

Inheritance – Achieved via struct embedding, where one struct embeds another to reuse its methods (similar to inheritance).

Abstraction – Achieved through interfaces, where the implementation details of a method are hidden, and only the method signature is exposed.

Goroutines are managed by the Go runtime.

OS threads are managed by the OS kernel.

The Go runtime schedules and maps goroutines onto OS threads using its internal scheduler.


Go runtime handles core detection: The Go runtime queries the OS at program startup to detect the number of CPU cores.

GOMAXPROCS defaults to the number of CPU cores: By default, Go will use the same number of CPU cores as are available on your system, but you can adjust it if needed.

Automatic core usage: Unless you specifically configure it, Go will automatically utilize all the available CPU cores for scheduling and executing goroutines.

Efficient scheduling: Go's scheduler runs a small number of OS threads (usually equal to the number of CPU cores) and multiplexes multiple goroutines on top of these threads, ensuring efficient concurrency management.

③ Interview-style explanation:

Goroutines are cheap because Go runtime handles them.

“Goroutines differ from threads because goroutines are scheduled by the Go runtime

📌 FOUNDATION TOPIC 4: Stack Overflow
① What is stack?

A stack stores:

function calls

variables

addresses

② What is stack overflow?

When the stack memory gets full, and the system can’t allocate more.

③ Why does it happen?

Too deep recursion

Too many local variables

Fixed-size stack limit

④ Why Go avoids it?

Traditional languages (C/Java/Python threads) have fixed stack sizes (~1MB).
If recursion grows → BOOM → stack overflow.

⑤ How Go avoids stack overflow?

Goroutines start with tiny stack (~2KB)

Stack grows dynamically as needed

Old stack copied → new bigger stack allocated

⑥ Interview-style explanation:

“In simple terms, Go prevents stack overflow using dynamic stack growth. Instead of starting each goroutine with 1MB stack like OS threads, Go starts with a few KB and expands as required. This gives memory efficiency and avoids stack overflow errors.”

⑦ Interview final answer:

“Go avoids stack overflow by using dynamically growing stacks for goroutines.”
